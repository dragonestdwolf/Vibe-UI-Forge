# 组件设计规范校验方案

## 整体前提
*   **已有设计稿 DSL 结构化数据**：包含所有图层、元素、样式、坐标、资源、变体属性、规范约束（颜色 / 圆角 / 间距 / 阴影 / 文本 / 图标规则等）。
*   **校验对象**：前端业务组件 / 基础组件。
*   **校验载体**：无头浏览器（Playwright/Puppeteer）渲染组件。
*   **核心逻辑**：不做 DOM 结构与设计稿图层强制层级映射，只做「设计稿标准属性」vs「组件渲染后真实计算属性」逐项比对。

## 一、总分权重结构（固定不变）

| 模块 | 权重/计分方式 | 说明 |
| :--- | :--- | :--- |
| **布局** | 错误倒扣分制 | 无固定占比，出现错误则扣分 |
| **资源调用** | 20% | 合规资源数 / 总资源数 × 20% |
| **样式 CSS 表比对** | 60% | 逐项属性比对计分 |
| **变体属性** | 20% | 合规属性数 / 总属性数 × 20% |
| **合计** | **量化满分：100%** | |

## 二、各模块校验流程 & 组件侧实现步骤

### 1. 布局模块（视觉截图比对 + 坐标校验｜错误倒扣分）

**校验项**
*   排版对齐（左 / 中 / 右 / 顶 / 底对齐）
*   文本错误换行、元素错位

**执行步骤**
1.  通过 Playwright/Puppeteer 渲染目标组件至画布。
2.  对组件整体进行全屏区域截图。
3.  从设计稿 DSL 中提取：元素基准坐标、文本行数、对齐基准线。
4.  组件侧通过 `getBoundingClientRect()` 获取所有元素、文本行真实坐标边界。
5.  逐一对齐基准、文本换行行数、元素相对位置做比对。
    *   **规则**：出现对齐错误、多余换行、元素偏移 → 按规则倒扣分；无错误不扣分。

### 2. 资源调用模块（占比 20%）

**计算公式**
$$ 	ext{得分} = \frac{	ext{合规资源数}}{	ext{总资源数}} 	imes 20\% $$

**校验项**
*   **Icon**：是否调用规范图标库、合规 Icon 数量占比
*   **图片**：是否符合规范风格、合规图片数量占比
*   **插画**：是否调用规范插画资源库、合规插画数量占比

**执行步骤**
1.  **采集资源**：渲染组件后，遍历 DOM 节点，批量采集三类资源：
    *   Icon 节点：自定义图标组件、`<svg>`、图标标签
    *   普通图片：`<img>`、背景图资源
    *   插画资源：指定插画容器节点
2.  **统计总数**：统计当前组件资源总数。
3.  **白名单校验**：
    *   **Icon**：组件名 / 类名 / 资源标识是否在规范 Icon 库白名单。
    *   **图片**：地址、风格、尺寸是否符合设计规范定义。
    *   **插画**：资源路径、标识是否在规范插画库白名单。
4.  **计算得分**：统计合规资源正确数，按公式计算本模块得分。

### 3. 样式 CSS 表比对模块（占比 60%）

**核心原理**
从设计稿 DSL 取出标准样式属性；组件侧通过浏览器 `getComputedStyle()` 取出渲染后最终真实计算样式。**无视 DOM 嵌套层级、多层实现，只比对最终视觉属性。**

**校验子项 & 执行通用步骤**
1.  从 DSL 解析出当前组件所有视觉元素的标准样式集。
2.  在渲染后的组件中，通过选择器定位关键视觉节点（根容器、轨道、填充、滑块、文本、图标、卡片等，**无需遍历全部 DOM**）。
3.  对每个关键节点执行 `getComputedStyle()` 获取浏览器最终计算样式。
4.  按以下分类逐项字段比对，统计合规属性占比，折算 60% 分值。

#### 3.1 阴影校验
*   **比对字段**：阴影颜色、透明度、模糊值、Y 轴偏移、X 轴偏移、扩展值

#### 3.2 描边校验
*   **比对字段**：描边粗细、描边颜色

#### 3.3 颜色校验
*   **比对规则**：颜色使用场景合规性（字体只用规范字体色、描边只用规范描边色、填充只用规范填充色）。
*   **统计**：合规填充色数量 / 总填充色数量

#### 3.4 圆角校验
*   **比对字段**：统一圆角 / 四角独立圆角
*   **统计**：合规圆角数 / 总圆角配置数

#### 3.5 间距校验（错误倒扣分 + 占比计分）
*   **比对维度**：
    *   内边距 Padding：上 / 下 / 左 / 右 单独值
    *   外边距 Margin
    *   横向元素间距 gap
    *   纵向元素间距
*   **执行方式**：
    1.  计算样式读取 padding/margin/gap。
    2.  元素真实坐标计算两两之间实际间距。
    3.  与 DSL 标准间距比对。
    *   **规则**：间距错误执行倒扣分；合规项纳入占比计分。

#### 3.6 文本校验
*   **比对字段**：字体家族、字号、字重

**本模块计分规则**
所有样式子项逐条比对，统计合规属性数 / 总样式属性数 →
$$ 	ext{得分} = 	ext{合规占比} 	imes 60\% $$

### 4. 变体属性模块（占比 20%）

**计算公式**
$$ 	ext{得分} = \frac{	ext{合规属性数}}{	ext{总属性数}} 	imes 20\% $$

**校验项**
*   组件变体属性、属性值拉表逐项比对（如 `size` / `variant` / `disabled` / `type` 等）

**执行步骤**
1.  从设计稿 DSL 提取组件标准变体属性表（属性名 + 标准值）。
2.  组件侧渲染时，采集组件实际入参 Props、静态 class 语义标识。
3.  构建实际属性对照表，与 DSL 标准属性表逐行比对。
4.  统计：正确匹配属性数 / 总变体属性数。
5.  按公式计算本模块得分。

## 三、关键问题解决方案（适配多层 DOM 组件如 Slider）

*   **背景**：设计稿 DSL 是扁平化视觉元素定义，前端组件是多层 DOM 实现。
*   **策略**：不强制做「DOM 节点和设计稿图层一一映射」。
*   **执行**：
    *   仅针对 DSL 定义的**关键视觉元素**。
    *   在组件中定位**对应关键渲染节点**。
    *   一对一拿计算样式比对。
    *   **中间嵌套层级、布局辅助 DOM 直接忽略**，不参与样式比对，避免层级不匹配导致误判。

## 四、总分合成规则

1.  **基础分** = 资源得分 + 样式得分 + 变体得分
2.  **布局模块**按错误项额外倒扣分
3.  **最终得分** = 基础分 - 布局扣分值

**输出结果**
*   组件规范符合度百分比
*   各模块明细合规率
*   错误问题清单
