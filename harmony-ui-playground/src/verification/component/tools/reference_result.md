# 组件设计规范校验方案

## 整体前提

- **已有设计稿 DSL 结构化数据**：图层、样式、坐标、资源、变体属性、规范约束（颜色 / 圆角 / 间距 / 阴影 / 文本 / 图标等）。
- **校验对象**：前端业务组件 / 基础组件。
- **校验载体**：无头浏览器（Playwright / Puppeteer）渲染组件。
- **核心逻辑**：不做 DOM 结构与设计稿图层的强制一一映射；对**关键视觉节点**做「设计稿标准属性」与「渲染后计算样式」逐项比对。

---

## 规范与工具文件来源（本仓库内路径）

以下路径均相对于 `harmony-ui-playground` 根目录。

| 校验类别 | 规范 / 数据文件 | 说明 |
| :--- | :--- | :--- |
| **阴影**（投影 / 模糊等） | `src/verification/component/css_styles_specification/shade.json` | Pixso EFFECT 等导出 |
| **描边** | `src/verification/component/css_styles_specification/outline.json` | 描边粗细、颜色等规范取值 |
| **颜色** | `src/verification/component/css_styles_specification/color.json` | 色板名称与色值；比对时需与 DSL 侧命名一致（见下「颜色命名」） |
| **圆角** | `src/verification/component/css_styles_specification/rounded.json` | 统一圆角 / 四角独立圆角等 |
| **间距** | `src/verification/component/css_styles_specification/gap.json` | Padding / Margin / gap 等间距 token |
| **文本** | `src/verification/component/css_styles_specification/font.json` | 字体家族、字号、字重等 |
| **默认渲染后排除的非关键 CSS** | `src/verification/component/tools/non_critical_css_properties_exclude_list.xlsx` | 不参与计分或比对的属性白名单；减少实现细节噪声 |

实现校验脚本时，应以对应 JSON 为**允许值 / 命名 / 语义**的权威来源；排除列表在聚合「待比对属性集合」时过滤字段。

## 一、总分权重结构（固定不变）

| 模块 | 权重 / 计分方式 | 说明 |
| :--- | :--- | :--- |
| **布局** | 错误倒扣分制 | 无固定占比，出现错误则扣分 |
| **资源调用** | 20% | 合规资源数 / 总资源数 × 20% |
| **样式 CSS 表比对** | 60% | 逐项属性比对计分 |
| **变体属性** | 20% | type 变体覆盖数 / type 变体总数 × 20% |
| **合计** | **量化满分：100%** | |

## 二、各模块校验流程与组件侧实现步骤

### 1. 布局模块（视觉截图比对 + 坐标校验｜错误倒扣分）

**校验项**

- 排版对齐（左 / 中 / 右 / 顶 / 底对齐）
- 文本错误换行、元素错位

**执行步骤**

1. 通过 Playwright / Puppeteer 将目标组件渲染到画布。
2. 对组件整体或约定区域截图。
3. 从设计稿 DSL 提取：元素基准坐标、文本行数、对齐基准线等。
4. 组件侧用 `getBoundingClientRect()` 等获取元素与文本行的真实边界。
5. 逐一对齐基准、换行行数、相对位置比对；出现对齐错误、多余换行、明显偏移 → 按规则倒扣分；无错误不扣分。

---

### 2. 资源调用模块（占比 20%）

**得分公式**

**得分** = (合规资源数 / 总资源数) × 20%

**校验项**

- **Icon**：是否调用规范图标库、合规数量占比
- **图片**：风格、尺寸、地址是否符合规范
- **插画**：是否在规范插画库白名单

**执行步骤**

1. **采集资源**：遍历 DOM，采集 Icon（自定义图标组件、`<svg>` 等）、`<img>` / 背景图、插画容器等。
2. **统计总数**。
3. **白名单校验**：与规范库标识、路径、风格规则对照。
4. **计分**：按上式计入本模块得分。

---

### 3. 样式 CSS 表比对模块（占比 60%）

#### 3.0 核心原理

1. **期望值来源**：从设计稿 DSL（及与 DSL 对齐的 token 名）解析出**当前场景下**各关键视觉元素的标准样式集；并与 `css_styles_specification` 下各 JSON 对齐（允许值、命名、语义）。
2. **实际值来源**：在渲染结果上定位**关键视觉节点**，对该节点调用 `getComputedStyle()`，得到浏览器**已级联、已解析**的最终计算值。
3. **比对原则**：**无视 DOM 嵌套层级与中间实现细节**，只比对「该节点最终呈现」上与规范相关的属性；与排除列表（`non_critical_css_properties_exclude_list.xlsx`）中的字段可跳过比对或不计入分母。


#### 3.1 阴影校验

- **规范来源**：`css_styles_specification/shade.json`
- **比对字段**：阴影颜色、透明度、模糊、X/Y 偏移、扩展（及规范文件中定义的等价字段）

#### 3.2 描边校验

- **规范来源**：`css_styles_specification/outline.json`
- **比对字段**：描边粗细、描边颜色（及规范中扩展字段）

#### 3.3 颜色校验

- **规范来源**：`css_styles_specification/color.json`
- **比对规则**：
  - **场景合规**：字体色 / 描边色 / 填充色等仅使用规范允许的颜色语义（与 `color.json` 中定义一致）。
  - **名称完全比对**：凡在 DSL、规范或组件侧以**命名 token**（设计变量名、资源名等）表达的，**CSS 侧使用的颜色命名**须与 **规范 `color.json` 中的 `name`** 以及 **DSL 校验数据里给出的颜色命名**三者一致（同一语义同一字符串）；禁止仅比对「算出来相近的 hex」而忽略命名不一致（除非业务明确仅验色值、不验 token，本方案默认**验名 + 验值**按规范执行）。
- **统计示例**：合规填充色数量 / 总填充色数量（分母分子定义与产品一致即可）

#### 3.4 圆角校验

- **规范来源**：`css_styles_specification/rounded.json`
- **比对字段**：统一圆角、四角独立圆角等
- **统计**：合规圆角数 / 总圆角配置数

#### 3.5 间距校验（错误倒扣分 + 占比计分）

- **规范来源**：`css_styles_specification/gap.json`
- **比对维度**：Padding（上 / 下 / 左 / 右）、Margin、横向 / 纵向 `gap` 等
- **执行方式**：
  1. 从计算样式读取 `padding` / `margin` / `gap` 等；
  2. 可用元素真实坐标推导相邻元素间净距，与 DSL 标准间距对照；
  3. 间距错误可倒扣分；合规项纳入占比计分。

#### 3.6 文本校验

- **规范来源**：`css_styles_specification/font.json`
- **比对字段**：字体家族、字号、字重（及 `font.json` 中约定字段）

#### 3.7 非关键 CSS 排除

- **列表文件**：`tools/non_critical_css_properties_exclude_list.xlsx`
- **用途**：默认渲染后应排除的属性（如部分布局辅助、浏览器默认噪声等），在生成「待比对属性列表」时剔除，避免误判；**不排除**仍需要从 DSL 取期望值：排除只影响**哪些键参与比对**，不替代「标准样式从哪来」。

**本模块计分**

**得分** = (合规样式属性数 / 总参与比对的样式属性数) × 60%

其中：

- **总参与比对的样式属性数（分母）**：组件在所有校验场景下，从 `getComputedStyle()` 枚举得到的**全量 CSS 属性条目**中，剔除 `non_critical_css_properties_exclude_list.xlsx` 后，且存在期望值映射（DSL / 规范）的属性总数。
- **合规样式属性数（分子）**：上述分母集合中，同时满足「规范允许值 / 命名 / 语义」与「组件设计稿 DSL 期望值」的属性条数。

---

### 4. 变体属性模块（占比 20%）

**得分公式**

**得分** = (type 变体覆盖数 / type 变体总数) × 20%

**计分口径（强制）**

- 本模块的 20% 分值，统一按 `typeVariantCount` 口径计分（业务形态变体覆盖率）。
- `type 变体总数`：来自渲染统计中的 type 维度总变体数（例如 `verification_result.json.render.typeVariantCount`）。
- `type 变体覆盖数`：成功渲染并进入 `verification_key_styles.xlsx` 主表（`统计结果`）的 type 变体数。
- 评分示例：`typeVariantCount=9` 且 9 个 type 均成功渲染并纳入主表，则本模块记为 `9/9`，得分 `20`。

**非计分但可输出的辅助口径**

- `stateVariantCount`（如 enabled/hover/focus/active/disabled）与 `seekbarVariantCount` 可作为诊断信息输出；
- 默认不纳入 20% 变体模块分子/分母，避免与 type 口径混算导致报告歧义。

**执行步骤**

1. 从 DSL 提取 type 变体清单（业务形态维度）。
2. 渲染侧按 type 变体逐一执行并生成主表记录。
3. 统计 `type 变体覆盖数 / type 变体总数` 并计分。
4. 若需附加输出 state/seekbar 口径，放在“统计对账/附录”中，不影响本模块分值。

## 三、多层 DOM 与 Slider 类组件（关键问题）

- **背景**：DSL 常为扁平视觉元素；前端是多层 DOM。
- **策略**：不强求「每个 DOM 节点对应每个设计图层」。
- **执行**：仅针对 DSL 中的**关键视觉元素**，在组件中定位**对应关键渲染节点**，一对一取 `getComputedStyle()` 比对；中间布局辅助节点可忽略，避免层级不一致导致误判。

---

## 四、期望值、计算样式与排除列表（规范性流程）

1. **期望值集合**：从设计稿 DSL（及与 DSL 对齐的 token 名）解析当前场景下各关键视觉元素的**标准样式键值**；允许值、命名、语义与 `css_styles_specification` 下各 JSON 对齐。若仅有排除列表而无 DSL（或等价黄金 JSON），无法判定对错。
2. **实际值集合**：在渲染结果上定位关键视觉节点，对该节点调用 `getComputedStyle()`，得到浏览器级联后的最终计算值；**不**以「仅遍历 DOM 收集声明过的 CSS」替代计算样式（声明与级联结果可能不一致）。
3. **参与比对的键**：在「期望键 ∪ 规范约定需校验的键」与计算样式对齐后，按 `non_critical_css_properties_exclude_list.xlsx` **剔除**不参与比对或不计分母的属性；剔除只影响**哪些键参与比对**，不改变「期望值必须从 DSL（或黄金表）来」的要求。
4. **比对执行**：对保留键逐项比对期望与实际；记录差异类型（缺失期望、值不一致、命名不一致等）供输出与计分使用。

## 五、总分合成规则

1. **基础分** = 资源得分 + 样式得分 + 变体得分  
2. **布局模块**按错误项额外倒扣分  
3. **最终得分** = 基础分 − 布局扣分  

### 输出结果（固定仅 2 个文件）


**文件 1：验证报告（Markdown）**

- 文件形态：`verification_result.md`。
- 必含内容：
  - 组件最终得分（含计算公式与分母/分子定义说明）。
  - 分模块结果（布局 / 资源 / 样式 / 变体）的合规率与得分。
  - 关键不符合项摘要（问题类型、影响范围、严重级别）。
  - 与 Excel 明细的统计对账值（总条数、参与计分条数、合规条数、不合规条数、排除条数）。
  - 当变体模块启用 type 口径时，需明确写出：`变体合规条数（按 typeVariantCount）= x / y`。

**文件 2：统计结果表（Excel，`.xlsx`）**

- 文件形态：`verification_key_styles.xlsx`。
- 目标：输出**可读、可对账、可评分**的统计结果，避免把全量 CSS 细节全部摊平成海量行。
- 默认采用**两层输出**（通用于所有组件，不是 Slider 专用）：
  1. `统计结果`（必选）：按「组件 + 变体 + 场景」聚合后的评分表（主交付）。
  2. `问题明细`（可选）：仅保留 `不符合` 项（用于追踪定位），不再输出“所有符合项”的全量属性明细。

### 2.1 Sheet：`统计结果`（主表，强制）

- **行粒度（强制）**：每行对应一条「组件 + 变体」聚合记录。
- **计分绑定（强制）**：变体模块 20% 的分子/分母，默认从本 Sheet 的 type 变体覆盖结果聚合得到（与 `typeVariantCount` 对账）。
- **推荐列（与业务示例一致，可按组件扩展）**：
  - `组件(变体）预览`
  - `样式-描边`
  - `样式-颜色`
  - `样式-圆角`
  - `样式-文本字体`
  - `样式-文本字号`
  - `样式-文本字重`
  - `样式-间距`
  - `阴影-模糊`
  - `阴影-Y轴偏移`
  - `阴影-X轴偏移`
  - `阴影-扩展`
  - `合计详情(颜色/圆角/间距/阴影/文本)`

### 2.2 Sheet：`问题明细`（可选，但建议保留）

- **行粒度**：每行一条不符合项（仅 `不符合`）。
- **建议列**：
  - `组件名称`、`变体标识`
  - `检查类别`（布局/资源/颜色/圆角/间距/阴影/文本）
  - `实际值`、`期望值`
  - `不符合类型`（值不一致/命名不一致/不在允许集合/缺少期望映射/来源不可追溯）
  - `不符合说明`
  - `规范来源`、`DSL来源`
  - `样式来源文件`、`样式来源定位`、`样式来源片段`

### 2.3 降噪约束

- 禁止输出“全量 computed style 全字段明细”作为主结果。
- 仅以下场景进入 `问题明细`：不符合、无法判定、来源不可追溯。
- 允许在 `问题明细` 中保留必要上下文，但主评分必须以 `统计结果` Sheet 为准。

## 六、变体关键样式清单与不符合统计（强制）

为避免只给总分、缺少可改造信息，所有组件验证结果需补充以下内容：

### 6.1 按变体输出关键样式（强制）

- 对组件的**每个业务变体**（例如 9 个或更多）分别输出关键样式清单。
- 清单需至少覆盖以下类别（无该类别时标记 `/`）：
  - 描边（宽度、颜色、样式）
  - 颜色（填充、文本、图标、背景等）
  - 圆角（统一圆角与四角）
  - 文本（字体家族、字号、字重、行高）
  - 间距（padding / margin / gap）
  - 阴影（颜色、x/y、blur、spread）
- 每条关键样式必须可追溯到：
  - 组件实现侧样式来源（CSS/TSX 文件与定位）
  - DSL 侧期望来源（组件 DSL）
  - 规范侧来源（`css_styles_specification/*.json`）

### 6.2 不符合项统计（强制）

- 在“变体维度”给出统计：每个变体的参与校验条数、合规条数、不合规条数、合规率。
- 在“类别维度”给出统计：描边/颜色/圆角/文本/间距/阴影各自的不合规条数与占比。
- 在“来源维度”给出统计：
  - 不符合 `css_styles_specification`（规范 DSL）的条数
  - 不符合组件 DSL（设计稿 DSL 期望值）的条数
- 不符合清单中需列出属性名、期望值、实际值、不符合类型与来源定位，保证可复核。

### 6.3 输出约束

- 报告中必须有“按变体关键样式清单”与“不符合统计”两个独立小节。
- 统计结果应可与 `verification_result.json` 对账；若有 Excel，也应与 Excel 对账。
- 该要求是**通用规则**，禁止在规范文档中写死某个具体组件名称或专属字段。
- 同一份报告中，变体模块在「计算公式 / 分模块表格 / 统计对账 / JSON 摘要」必须使用同一分母口径（默认 `typeVariantCount`），禁止混用 type 与 state 口径。
