# 03 | 长期目标

> 这一篇说明 Design System Skill 的愿景、目标用户和产品定位。

---

## 目标用户

**你是一位平台设计师，负责维护资源层。**

你的工作是把设计资源（Figma / Pixso）转换为 AI 大模型可以理解的语料，让 AI 能够根据设计规范生成符合预期的 UI 页面。

你不一定熟悉 React、TypeScript 等开发细节，但你需要了解：
- 本项目的架构是怎样的
- 文件之间如何互相调用
- 你需要维护哪些内容
- 这些内容应该如何被制造和转化

---

## 资源层维护者的职责

你的核心工作是把**设计工具中的设计资源**转换为** AI 可以理解和使用的语料**。

这个转换过程通常包括：

| 输入 | 输出 | 你的工作 |
|------|------|----------|
| Figma / Pixso 设计稿 | 设计系统资源 | 从设计稿中提取组件、样式、布局 |
| 设计稿中的组件变体 | `.resources/` 中的配置 | 描述组件的属性、状态、用法 |
| 设计稿中的页面布局 | `blocks.json` 中的 Block 模板 | 定义页面的典型结构 |
| 设计系统规范 | `route-index.md` / `layout/*.md` | 将设计意图转化为 AI 可读的文档 |

简而言之：**你负责喂料，AI 负责生成。**

---

## 我们要解决什么问题？

### 现状痛点

设计师交付设计稿 → 开发者手工还原 → 容易出现偏差 → 反复沟通 → 效率低

### 我们的目标

设计师 → AI 生成 + 设计师审核 → 快速还原设计 → 减少沟通成本

---

## 你需要理解什么？

通过阅读本 Wiki，你应该掌握：

### 1. 项目架构

三层架构（Agent Layer / Resource Layer / Source Layer）如何分工协作。

详见：[01-架构总览](./01-架构总览.md)

### 2. 文件之间的调用关系

当你维护某个文件时，知道它会被谁读取、会影响什么。

详见：[02-完整架构概览](./02-完整架构概览.md)

### 3. 你需要维护什么

- `.resources/` 下的配置和索引文件
- `route-index.md` - 路由索引
- `layout/*.md` - 页面布局模板
- `blocks.json` - Block 页面模板索引
- `components.json` - 组件索引

### 4. 内容如何被制造出来

你通过和 Agent 交互，把 Figma/Pixso 上的设计稿转化为上述文件。

这个过程不需要你手写代码，而是告诉 AI 你看到了什么设计稿，AI 帮你生成对应的配置。

详见：[03-资源层替换指南](../03-设计系统资源/03-资源层替换指南.md)

## 长期愿景

### 愿景一：设计师主导，AI 执行

你只需要描述设计稿中的内容，AI 帮你转换成可用的资源文件。

### 愿景二：设计即实现

设计稿中的样式、组件、布局，直接映射为 AI 可读的语料。

设计稿和最终页面保持一致，减少「设计稿很漂亮，实现走样」的问题。

### 愿景三：人人可用的设计系统

无论你用的是哪个设计系统（Harmony、DevUI、自定义），都可以用同一套流程接入和管理。

---

## 我们不是做什么的？

| 不是 | 说明 |
|------|------|
| 不是编程教程 | 你不需要手写代码 |
| 不是 Figma 替代品 | 设计稿还是在 Figma/Pixso 里做 |
| 不是全自动流水线 | 需要你和 AI 协作迭代 |
| 不是万能生成器 | 复杂定制仍需开发者介入 |
