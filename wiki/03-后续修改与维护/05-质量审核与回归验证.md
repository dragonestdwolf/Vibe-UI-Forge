# 05 | 质量审核与回归验证

> 这一篇说明资源接入或维护修改完成后，设计师和 AI 分别要检查什么。
> 承接 02 / 03 / 04 的修改结果。

---

## 开头三问

**什么时候需要走这一层？**

当 02 / 03 / 04 的修改完成后，都需要进入质量审核环节。修改完成不代表验收通过，必须跑完质量门禁并通过设计师最终验收。

**设计师只需要做什么？**

在 AI 报告质量门禁结果后，做最终视觉和功能验收（Storybook 检查）。然后判断 PASS / FAIL 并说明理由。

**Agent 会自动做什么？**

跑质量门禁三个命令、报告验证结果、写 generation log。

---

## 1. 质量门禁：正式验收标准

这是本项目所有修改的正式质量门禁。02 / 03 / 04 各章提到的验证均引用此标准，不再重复列出具体命令。

三个验证命令（来自 Skill 原生支持）：

```bash
node scripts/validate_design_system_resources.mjs   # 资源索引完整性验证
npm run build                                      # 项目构建验证
npm run build-storybook                            # Storybook 构建验证
```

| 命令 | 验证什么 | 通过标准 |
|---|---|---|
| `validate_design_system_resources.mjs` | blocks.json / components.json 的路径是否指向真实文件 | 无报错，路径全部 resolve |
| `npm run build` | TypeScript 类型、ESLint 规则 | 构建成功，无错误 |
| `npm run build-storybook` | Storybook story 渲染 | 构建成功，story 能正常展示 |

修改完成后必须全部通过，才能进入设计师验收。

---

## 2. 两种验收场景

| 场景 | 验收重点 |
|---|---|
| **新增资源**（Component / Block / Page） | 新增内容是否完整注册、story 是否覆盖、markdown 是否写好、资源路径是否正确 |
| **修改资源** | 源码和 markdown 是否同步、接口是否向后兼容、影响范围是否已说明 |

---

## 3. 完整验收工作流

```
🤖 AI 完成修改（02 / 03 / 04 各层的修改）
    ↓
🤖 AI 跑质量门禁三个命令
    ↓
🤖 AI 说明验证结果（通过 / 失败 / 部分失败）
    ↓
👤 设计师 做最终视觉和功能验收
    ↓
🤝 双方确认 PASS / FAIL
    ↓
FAIL → 返回修改阶段
PASS → 记录到 generation log
```

---

## 4. 新增资源验收清单

新增 Component / Block / Page 时，设计师检查：

**注册完整性**

- [ ] 新组件已注册到 `components.json`（Component 新增）
- [ ] 新 Block 已注册到 `blocks.json`（Block 新增）
- [ ] 新 page_type 已注册到 `route-index.md`（Page 新增）
- [ ] 新 layout markdown 已写好（Page 新增）

**质量门禁**

- [ ] `validate_design_system_resources.mjs` 通过
- [ ] `npm run build` 通过
- [ ] `npm run build-storybook` 通过

**设计验收**

- [ ] 视觉符合设计稿
- [ ] 状态覆盖完整（默认、禁用、选中、错误等）

---

## 5. 修改资源验收清单

修改已有 Component / Block / Page 时，设计师检查：

**一致性**

- [ ] 源码改了，markdown 同步更新了（block/*.md 或 layout markdown）
- [ ] blocks.json / components.json 的路径仍然有效

**接口兼容性**

- [ ] 已有 Props 未被删除或重命名（Component 修改）
- [ ] 组件嵌套关系未发生变化（Block 修改）
- [ ] layout_skeleton 结构未被破坏（Page 修改）

**质量门禁**

- [ ] `validate_design_system_resources.mjs` 通过
- [ ] `npm run build` 通过
- [ ] `npm run build-storybook` 通过

**影响范围**

- [ ] AI 已说明影响哪些已有组件或页面
- [ ] 影响范围可接受

**设计验收**

- [ ] 视觉符合设计稿
- [ ] 状态覆盖完整

---

## 6. PASS / FAIL 判断标准

| 情况 | 判断 | 处理 |
|---|---|---|
| `validate_design_system_resources.mjs` 失败 | **FAIL** | 资源索引路径有问题，必须修复 |
| `npm run build` 失败 | **FAIL** | TypeScript 或构建错误，必须修复 |
| `npm run build-storybook` 失败 | **FAIL** | story 渲染问题，必须修复 |
| 源码改了，markdown 没改 | **FAIL** | 违反同步原则，必须补 markdown |
| 已有 Props 被删除 | **FAIL** | 违反向后兼容，必须回滚 |
| 组件嵌套关系被破坏 | **FAIL** | 违反 Block 接口稳定，必须回滚 |
| layout_skeleton 被破坏 | **FAIL** | 违反 Page 接口稳定，必须回滚 |
| 视觉不符合设计稿 | **FAIL** | 返工 |
| 质量门禁全部通过 + 视觉符合 | **PASS** | 记录到 generation log |

---

## 7. generation log 格式

修改完成后写入：

```markdown
## 原始需求
[用户原始请求或设计稿链接]

## Route Matching（如涉及页面生成）
page_type: xxx

## Layout Resolution（如涉及页面生成）
layout: layout/xxx.md

## Source Grounding
reference_blocks: xxx
needed_components: xxx
修改的文件:
  - src/component/xxx/index.tsx
  - .resources/harmony/block/xxx.md

## 质量门禁验证
validate_design_system_resources.mjs: 通过 / 失败
npm run build: 通过 / 失败
npm run build-storybook: 通过 / 失败

## 影响范围
[AI 说明影响了哪些已有组件或页面]

## 设计师验收
验收结果: PASS / FAIL
需要返工的问题: [如有]
可以沉淀的经验: [如有]
```

---

## 8. 设计师给 AI 的推荐开头

```text
请跑完质量门禁（validate_design_system_resources.mjs + build + build-storybook），
然后报告验证结果。
```

如果验收 FAIL：

```text
[具体问题] 不符合预期，需要修复。
```
