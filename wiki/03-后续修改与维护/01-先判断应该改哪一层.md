# 01 | 先判断应该改哪一层

> 维护时最重要的不是马上改，而是先判断问题属于 CSS / Token、Component，还是 Page / Block。

---

## 开头三问

**什么时候改这一层？**

这是入口章节，不需要判断"什么时候改01层"。读完这一篇后，才能判断接下来的 02 / 03 / 04 / 05 谁来处理你的需求。

**设计师只需要做什么？**

说清楚"这个修改影响什么"。不需要判断具体要改哪个文件，只需要说清楚你看到的视觉变化、组件行为变化或页面结构变化。

**Agent 会自动做什么？**

AI 会分析需求，判断属于哪一层（CSS / Token / Component / Page / Block），说明影响范围，并在对应层执行修改。

---

## 1. 先建立维护判断框架

### 1.1 为什么要先判断层级

修改前先判断属于哪一层，能避免两个最常见的错误：

- **改错地方**：页面布局问题被误判成组件问题，改完问题依然复现
- **改多了**：只是想换个颜色，却把组件结构也改了，意外影响其他页面

判断层级不需要写代码，只需要说清楚"这个修改影响什么"。

### 1.2 三个主要修改层

维护和业务主题接入时，涉及改动的通常只有三层：

| 层级 | 改的是什么 | 对应 Source Layer 目录 |
|---|---|---|
| **CSS / Token** | 颜色、圆角、字体、间距等视觉值 | `src/styles/` |
| **Component** | 组件的 variant、slot、Props、交互状态 | `src/component/{ComponentName}/` |
| **Page / Block** | 页面结构、区块组合、新增页面类型 | `src/blocks/` + Resource Layer |

### 1.3 Source Layer 和 Resource Layer 的关系

日常维护主要改 Source Layer（`src/styles/`、`src/component/`、`src/blocks/`）。

Resource Layer 什么时候需要改？只有当页面类型、Layout 结构或 Block 注册发生变化时：

| Resource Layer 文件 | 什么情况下需要改 |
|---|---|
| `route-index.md` | 新增 `page_type` 或 `hit_rules` 发生变化 |
| `layout/{page_type}.md` | 页面骨架（layout_skeleton）变化 |
| `blocks.json` | Block 源码路径变化，或新增了 Block |
| `block/*.md` / `component/*.md` | Block 或 Component 的使用说明需要更新 |

换句话说：**日常改视觉和组件实现主要在 Source Layer；改完 Source Layer 后，如果 page_type、layout、Block 注册也跟着变了，Resource Layer 必须同步更新。**

## 2. 两类设计师

| 角色 | 职责范围 | 典型工作 |
| --- | --- | --- |
| **平台设计师** | 基于平台提供的组件库做页面配置和微调，不改底层组件源码 | 调整页面布局、组合已有 Block、配置组件 props |
| **业务设计师** | 在 Source Layer 做组件变体修改、样式调整、区块新增，需要了解 Resource Layer 索引 | 修改组件 variant、新增样式 token、创建新 Block |

```
平台设计师
  └── 使用已有的 Component / Block / layout 组合页面
  └── 不修改 src/component/ 源码
  └── 不修改 token

业务设计师
  └── 在 src/component/ 中修改组件变体或样式
  └── 在 src/styles/ 中调整 token
  └── 可能需要修改 Resource Layer 索引
```

## 3. 如何判断应该改哪一层

### 3.1 三层边界

| 需求 | 应该进哪一层 | 改哪些文件 |
| --- | --- | --- |
| 改颜色、圆角、字体、间距 | CSS / Token | `src/styles/*.css`（优先）、`src/index.css`（仅在必要时） |
| 新增组件 variant、slot、结构、交互 | Component | `src/component/{ComponentName}/index.tsx` |
| 调整页面布局、区块顺序、新增 page_type | Page / Block | `src/blocks/*.tsx` + Resource Layer（如涉及 layout 变化） |

### 3.2 判断流程

```
这个需求只是视觉变化吗？（颜色、间距、字号、圆角）
  是 → CSS / Token 层
  否
    ↓
它是否改变单个组件的 API、结构或行为？
  是 → Component 层
  否
    ↓
它是否改变页面结构、区块组合或页面类型？
  是 → Page / Block 层
  否
    ↓
这个修改会影响其他页面吗？
  是 → 可能跨层，先说明影响范围
```

### 3.3 快速判断表

| 需求 | 判断结果 |
| --- | --- |
| 把按钮的 primary 颜色从蓝色改成绿色 | CSS / Token |
| 给 Switch 组件新增一个 disabled 状态 | Component |
| 设置页新增一个"关于我们"区块 | Page / Block |
| 修改健康仪表盘三叶草的顺序 | Page / Block |
| 所有设置页的标题字号统一改大 | CSS / Token |
| 平台设计师调整某个页面的 Block 组合 | Page / Block |
| 业务设计师给 Button 新增一个 loading variant | Component |

## 4. 完整维护工作流

```
业务设计师 / 平台设计师 提交修改需求
    ↓
🤝 双方对齐：修改范围、影响面、是否跨层
    ↓
🤖 AI 判断属于哪一层：CSS / Token / Component / Page-Block
    ↓
┌─ CSS / Token 层 ──────────────────┐
│  🤝 设计师确认修改范围和视觉预期      │
│  🤖 AI 修改 src/styles/            │
│  👤 设计师在 Storybook 验收        │
└────────────────────────────────────┘
    ↓
┌─ Component 层 ─────────────────────┐
│  🤝 设计师说明组件变体 / 状态变化     │
│  🤖 AI 修改 src/component/         │
│  👤 设计师在 Storybook 验收        │
└────────────────────────────────────┘
    ↓
┌─ Page / Block 层 ─────────────────┐
│  🤝 设计师说明页面结构变化           │
│  🤖 AI 修改 src/blocks/            │
│  🤖 AI（如需要）同步更新 Resource Layer │
│  👤 设计师验收页面结构              │
└────────────────────────────────────┘
    ↓
🤖 AI 跑质量门禁验证
    ↓
🤖 AI 写 Generation Log（如涉及页面生成）
```

## 5. 为什么要先判断层级

如果层级判断错，常见后果是：

| 错误判断 | 后果 |
| --- | --- |
| 只是改颜色，却改了组件结构 | 污染组件 API，影响所有用到这个组件的地方 |
| 只是页面特殊需求，却改了全局 Button | 意外影响其他页面 |
| 页面布局问题被误判成组件问题 | 改错地方，问题复现 |
| 局部 render 结果被误当成稳定模板 | `src/render/` 是生成预览，**不能作为模板源** |

## 6. 设计师给 AI 的推荐开头

```text
请先判断这个需求属于哪一层：CSS / Token、Component、还是 Page / Block。
如果你认为需要跨层改动，请先说明原因和影响范围。
```

平台设计师可以说：

```text
这是平台设计师的页面配置修改，目标是组合已有的 Block，不改底层组件源码。
请判断属于哪一层。
```

业务设计师可以说：

```text
这是业务设计师的组件变体 / 样式调整，需要修改 Source Layer。
请判断属于哪一层，如果涉及 Resource Layer 更新请说明。
```
