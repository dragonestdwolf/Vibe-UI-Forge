# 09 · 改 Page 与 Block 的工作流

> 当需求已经不是单个组件问题，而是页面结构、区块组合、信息层级变化时，应该在 page / block 层做改造。

---

## 这类改造解决什么问题

页面级别的改造需求 — 布局结构变化、区块重新组合、信息层级调整、超出了组件层面的改造范畴。

---

## 什么时候应该进入这一层

| 需求 | 属于 Page / Block 层？ |
|------|----------------------|
| 登录页从单栏改成双栏 | ✅ |
| dashboard 区块重排 | ✅ |
| marketing hero 增加额外说明区 | ✅ |
| 列表页筛选区从侧边移到顶部 | ✅ |
| 页面结构不再适合现有 block | ✅ |
| 把按钮变成品牌主色 | ❌ → CSS / Token 层 |
| 给 Select 新增一个 variant | ❌ → Component 层 |

---

## Page / Block 和 Component 层的边界

| 如果你只是想…… | 应该先尝试 |
|----------------|-----------|
| 调整区块顺序、增删区块 | Page / Block 层 |
| 改页面局部间距、对齐方式 | CSS / Token 层（className）|
| 改变某个组件的结构或行为 | Component 层 |

**原则**：先判断这个需求是否必须通过页面/区块结构调整来实现，再决定进入这一层。

---

## Page / Block 层的三种处理方式

| 方式 | 适用场景 | 风险 |
|------|---------|------|
| **小改现有 block** | 改文案、换区块顺序、替换个别组件 | 低 |
| **重组 block** | 页面结构大改、block 内部层级明显变化 | 中 |
| **放弃 block，从 page 重新搭** | 需求与现有 block 差异过大，block 已经不能作为合理起点 | 高 |

**原则**：能用小改解决的，不做重组；能用重组解决的，不做重建。

---

## 标准工作流

```
① 判断：这是页面/区块结构问题，还是组件问题？
    ↓ 是页面/区块问题
② 判断：现有 block 能不能作为起点？
    ↓ 能小改
③ 选择：小改 / 重组 / 重建
    ↓
④ 实施
    ↓
⑤ 验证：是否引发了不该有的组件层改动
```

---

## AI 默认修改范围

AI 在 Page / Block 层的默认修改顺序：

1. 目标 page 文件
2. block 组合关系、section 顺序
3. 页面级文案、布局、栅格、容器宽度
4. 当前页面内的局部组件组合方式

**可联动**：页面级数据文件、文案配置文件。

**默认禁区**：除非明确授权，否则不回头重写基础 Button / Card / Input、不改全局 token、不顺手改其他页面模板。

---

## 本地项目改造时如何与 Agent 协作

### 设计师怎么说

```
这是本地项目页面改造，请直接改当前项目的 page 和局部 block，
不需要把它做成可分发模板。
```

### Agent 应该怎么做

1. 直接改目标 page 文件
2. 改这个页面对应的 block 组合
3. 改页面自带的局部组件
4. 本质是"把页面做对"，不是"做可安装模板"

---

## 可分发资源改造时如何与 Agent 协作

### 设计师怎么说

```
我要把这个页面方案做成可分发的 block / page 资源。
请按可安装模板的方式组织文件，并告诉我目标页面、附带组件和依赖项会如何落地。
```

### Agent 应该怎么做

1. 把页面模板拆成 `registry:block` / `registry:page` / `registry:component` 等可分发文件集合
2. 处理目标路径、依赖组件和附带文件
3. 让它可以被其他项目通过安装命令接入
4. 本质是"模板资产化"，不是单个页面交付

---

## 什么时候 AI 必须先确认

- 发现现有 block 已完全不适合作为起点，需要整页重建
- 为了完成页面目标，必须回头改基础组件（说明需求判断有误，可能应该先解决组件问题）
- 页面改造会联动多个模板或多个路由
- 页面视觉问题其实更适合回到 CSS / Component 层统一处理

---

## 给 AI 的推荐指令模板

```text
我要改什么：把 [具体页面/区块] 从 [现状] 改成 [目标]
这属于哪一层：Page / Block 层
优先改哪里：小改现有 block / 重组 block / 重建 page
不要碰什么：不要先改 Button、Input 这类基础组件，不要改全局 token
如果现有 block 差异太大，可以重组或重建，但请说明你选择的是哪一种
改完后怎么汇报：
  - 改了哪些 page / block 文件
  - 你走的是小改 / 重组 / 重建中的哪一种
  - 是否越级改到了 component 层
  - 重点验收哪里
```

## 一个真实案例

**需求**：把 login-01 从单栏登录页改成双栏，左侧放品牌说明，右侧保留登录表单。

**AI 执行**：
1. 判断：现有 block 可以作为起点，做小改
2. 改 block 布局：双栏 grid 结构
3. 不改 Input、Button 等基础组件
4. 汇报：改了哪些 block 文件，走的是"重组"路径

**结果**：一个 block 布局调整，登录页变成双栏，基础组件保持不变。

---

## 常见误区

| 误区 | 正确做法 |
|------|---------|
| 只想小改文案，却去改内部组件源码 | 应该在 page/block 层改文案，不碰组件 |
| 明明 block 已经不合适，还死撑着 patch | 判断是否应该重组或重建 |
| 页面问题误判成组件问题 | 先判断是否必须通过页面结构调整实现 |
| 为了页面效果去改全局 token | token 改全局，页面结构是另一层 |
| 顺手把其他页面也一起改了 | 一次只改目标页面，不扩散范围 |

---

## 验收清单

- [ ] 页面结构是否更接近预期（布局、区块顺序、信息层级）
- [ ] 版式是否稳定（刷新后是否保持一致）
- [ ] 是否为了页面局部需求引发了底层组件污染（基础组件被不必要地改了）
- [ ] AI 有没有越级改到 component 层（本来是 page 问题，结果顺手改了 Button）
- [ ] 改动是否只影响了目标页面，没有扩散到其他页面
- [ ] 如果走了"可分发路径"，安装落点、文件组织、依赖链是否完整
