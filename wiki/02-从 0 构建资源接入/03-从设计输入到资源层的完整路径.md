# 03 | 从设计输入到资源层的完整路径

> 这一篇是资源接入主线。真实输入不一定只有 Figma，也可能来自设计文档、截图、设计稿、旧项目代码，甚至 Vue / Angular 等框架里的已有组件。资源接入的目标，是把这些不同来源统一整理成同一套可被 Skill 消费的 React + Tailwind 架构。

---

## 1. 先理解资源接入

### 1.1 先用一句话理解

资源接入不是“把 Figma 转成代码”，而是把各种设计输入统一变成一套稳定的前端真值和资源地图。

```text
多种输入源
  ↓
统一成 React + Tailwind + shadcn registry-compatible 架构
  ↓
抽取 Component / Block / Page Type / Layout
  ↓
形成 Resource Layer
  ↓
让 Skill 稳定生成页面
```

当前 Harmony 示例项目使用的基准架构是：

| 能力 | 当前版本 |
|---|---|
| React | `^19.2.4` |
| Tailwind CSS | `^4.2.1` |
| shadcn | `^4.1.2` |
| Vite | `^7.3.1` |
| TypeScript | `~5.9.3` |
| Storybook | `^10.3.5` |
| Icon Library | `lucide-react` |

后续接入别的设计系统时，也应该先明确一套类似的目标架构和版本号。否则 AI、工程、设计师说的“组件”“样式”“状态”很容易不是同一件事。

---

### 1.2 输入源不只有 Figma

Figma / Pixso 只是最常见的输入。真实项目里，设计系统资料可能来自很多地方：

| 输入源 | 能提供什么 | 需要注意什么 |
|---|---|---|
| Figma / Pixso | 组件结构、变量、Auto-layout、截图、节点信息 | 它表达设计意图，不等于可运行真值 |
| 设计文档 | 使用场景、组件语义、状态说明、规范原则 | 需要转成可执行的组件 API 和验收标准 |
| 截图 / 静态设计稿 | 视觉参考、页面层级、布局比例 | 只能辅助还原，不能替代组件源码 |
| 旧 React 项目 | 已有组件、样式、交互和业务逻辑 | 需要整理成稳定组件、stories 和资源索引 |
| Vue / Angular 项目 | 旧组件语义、状态、样式和业务经验 | 需要迁移为 React 组件，不能直接作为 Skill 真值 |
| 组件库文档 | 组件说明、Props、变体、示例 | 要确认源码和文档是否一致 |

所以这一阶段更准确的名字是：

```text
从设计输入到统一组件架构，再到 Resource Layer
```

---

### 1.3 为什么要统一成 React + Tailwind 架构

AI 生成页面时，最怕的是每个输入源都有自己的表达方式：

- Figma 说的是 frame、auto-layout、variant。
- 设计文档说的是使用场景和规范语言。
- 截图只能看到结果，看不到组件 API。
- Vue / Angular 组件有自己的 props、slot、模板语法。
- 旧项目 CSS 可能有历史包袱和隐式依赖。

如果不统一，AI 每次生成都要临时翻译这些材料，结果就会不稳定。

统一成 React + Tailwind + shadcn registry-compatible 架构，是为了让所有输入最后落到同一组可验证对象上：

| 统一对象 | 作用 |
|---|---|
| React Component | 封装结构、Props、状态和交互 |
| Tailwind / CSS token | 统一颜色、间距、圆角、字体等视觉规则 |
| Storybook stories | 展示组件状态和变体，方便设计师验收 |
| `components.json` | 让 AI 知道组件在哪里、如何 import |
| `blocks.json` | 让 AI 知道哪些 Block 可以作为稳定模板 |
| `route-index.md` / `layout/*.md` | 让 AI 知道页面类型和布局规则 |

这套方式借用了 shadcn 的核心思路：组件源码直接存在项目里，`components.json` 描述项目配置和 alias，registry 工作流可以注册、分发和复用组件。

本项目不是只能接 shadcn 组件，但它需要兼容这种工作方式：**组件必须是可读、可运行、可注册、可验证的 React 源码。**

## 2. 资源接入主线

### 2.1 完整链路

```text
输入源收集
  ↓
设计师梳理语义：页面族、组件职责、状态、验收标准
  ↓
AI / 工程统一成 React Component + Tailwind token
  ↓
补齐 stories，确认组件状态和变体可检查
  ↓
从代表页面抽取稳定 Block
  ↓
归纳 Page Type，编写 route-index.md 和 layout markdown
  ↓
生成 components.json / blocks.json
  ↓
形成可被 Skill 消费的 Resource Layer
  ↓
用真实页面 prompt 试跑并验收
```

这条链路里，Figma MCP、Pixso MCP、截图识别、旧项目迁移都只是输入方式。真正的目标是后面的统一架构和资源层。

---

## 3. 组件导入工作流

### 3.1 每一步谁做什么

先记住三个标记：

```text
🤖 = Agent 独立完成
👤 = 设计师独立完成
🤝 = 设计师和 Agent 协作
```

这条流程的目标是：**Agent 尽量自动完成组件导入，设计师只负责给对输入、检查视觉、确认结果。**

设计师最重要的两件事：

- 确保 MCP 链接、节点 ID、截图或设计稿范围是正确的，Agent 能读到目标设计输入。
- 打开 benchmark HTML 或 Storybook，检查样式、状态和交互是否符合设计稿。

---

### 3.2 Benchmark HTML 预览

这一阶段先从组件 MCP 生成 benchmark HTML。它不是最终 React 组件，而是用来对齐设计稿的第一版可视化基准。

| 谁 | 做什么 |
|---|---|
| 🤝 设计师 + Agent | 设计师提供组件的 Figma / Pixso MCP 链接、节点 ID、截图或设计稿范围。 |
| 🤖 Agent | 读取设计输入，分析组件结构、布局、状态和变体，生成初版 benchmark HTML。 |
| 👤 设计师 | 打开 benchmark HTML，检查是否贴近设计稿：样式有没有还原、交互状态有没有列全、明显缺失的变体有没有漏掉。 |

设计师可以这样说：

```text
请根据这个组件 MCP 链接生成 benchmark HTML 预览。
先不要写 React 组件，先对齐组件结构、状态、变体和交互样式。
```

这一阶段设计师不需要写代码，只需要确认“标杆看起来对不对”。

---

### 3.3 React Component Generation

这一阶段把通过初步确认的 benchmark HTML 转成真实 React 组件源码。

| 谁 | 做什么 |
|---|---|
| 🤝 设计师 + Agent | 设计师确认 benchmark HTML 方向正确后，让 Agent 转成组件源码。 |
| 🤖 Agent | 基于 benchmark HTML 生成 `tsx` 组件、样式引用和必要的 Props / variants。 |
| 🤖 Agent | 将组件放到 `src/component/{ComponentName}/`，并补充 story，方便后续在 Storybook 里验收。 |
| 👤 设计师 | 通常不需要看源码，只需要知道组件会进入 Storybook 做视觉验收。 |

设计师可以这样说：

```text
benchmark HTML 方向没问题。
请把它转换成 React TSX 组件源码，并补充 Storybook story 用于审查样式和状态。
```

这一阶段 Agent 应该优先复用项目现有 token、Tailwind 规则、工具函数和组件组织方式，避免把 benchmark HTML 原样硬塞进 React。

---

### 3.4 Storybook Rendering & QA

这一阶段把 React 组件渲染到 Storybook，让设计师检查真实组件效果。

| 谁 | 做什么 |
|---|---|
| 🤖 Agent | 拉起 Storybook 或提供启动命令，并确保组件 story 能正常渲染。 |
| 🤖 Agent | 如果有明显样式问题、类型错误或 story 渲染失败，先自行修复再交给设计师验收。 |
| 👤 设计师 | 打开 HTML 或 Storybook 检查：颜色、间距、字号、圆角、状态切换、hover / active / disabled 是否符合预期。 |

设计师可以重点检查：

- 默认态、选中态、禁用态、加载态、错误态是否齐全。
- 颜色、间距、字号、圆角是否和设计稿一致。
- 交互状态是否能切换，状态命名是否符合设计语义。
- 是否有明显“看起来像，但不是同一个组件”的临时拼装。

在当前示例项目里，Storybook 通常由 Agent 在 `harmony-ui-playground/` 目录下拉起：

```bash
pnpm storybook
```

如果 Agent 已经拉起 Storybook，设计师只需要打开 Agent 提供的本地地址检视样式。

---

### 3.5 验收与修正

这一阶段根据设计师反馈做闭环。如果样式符合设计稿，就进入注册；如果不符合，就由 Agent 修正后再次渲染。

| 谁 | 做什么 |
|---|---|
| 👤 设计师 | 在 Storybook 中确认组件是否通过视觉和状态验收。 |
| 🤖 Agent | 如果 FAIL，根据反馈修正 TSX / CSS / stories，再重新拉起或刷新 Storybook。 |
| 🤖 Agent | 如果 PASS，整理组件源码、story、导出路径，并准备注册到资源层。 |

完成标准：

```text
benchmark HTML 对齐设计稿
  ↓
React TSX 组件生成完成
  ↓
Storybook 渲染通过
  ↓
设计师视觉验收通过
```

---

### 3.6 组件注册与使用说明

这一阶段把通过验收的组件正式录入资源层。完成后，这个组件才算可以被页面生成流程调用。

| 谁 | 做什么 |
|---|---|
| 🤖 Agent | 更新 `.resources/{system}/components.json`，注册组件 ID、源码路径、导出名、stories 路径。 |
| 🤖 Agent | 确认组件源码路径、story 路径和 import alias 都能被项目正确读取。 |
| 🤖 Agent | 如果组件很复杂，补充 `.resources/{system}/component/{component-id}.md`，说明 Props、变体、状态、使用场景和禁止组合。 |
| 👤 设计师 | 确认组件名称、语义、状态、变体和最终渲染结果都正确。 |

复杂组件建议补 usage markdown，例如：

```text
.resources/{system}/component/settings-row.md
```

这份 markdown 不是源码真值，而是给大模型按需读取的使用指导。页面生成时，如果模型不确定组件怎么用，可以请求并参考这份说明。

---

### 3.7 设计师最小介入清单

设计师不需要跟着每一步做工程操作，只需要守住下面几个关键点：

| 阶段 | 设计师需要做什么 |
|---|---|
| 输入前 | 提供正确的 MCP 链接、节点 ID、设计稿范围或截图 |
| Benchmark 后 | 看 benchmark HTML 是否贴近设计稿 |
| Storybook 后 | 打开 Storybook，检查颜色、间距、字号、状态切换和交互样式 |
| 注册前 | 确认组件名称、语义、状态和变体可以正式纳入系统 |

除此之外，组件分析、benchmark HTML 生成、React TSX 转换、story 补齐、样式修正、组件注册和复杂组件 usage markdown，都应该优先交给 Agent 自动完成。

## 4. 设计师关注点

### 4.1 设计师应该关注什么

设计师不需要手写复杂 JSON，也不需要决定 React 代码怎么组织。设计师最重要的职责是把“设计语义”和“验收标准”说清楚。

| 设计师关注 | 需要说明什么 |
|---|---|
| 页面族 | 哪些页面属于同一类，例如设置页、仪表盘、列表页 |
| 页面类型 | 这一类页面应该叫什么，和哪些页面相似或不同 |
| 组件职责 | 哪些是基础组件，哪些是业务组件，分别解决什么问题 |
| 状态和变体 | 默认、选中、禁用、加载、错误、空状态等 |
| 组合规则 | 哪些组件可以放在一起，哪些组合不应该出现 |
| 视觉优先级 | 哪些颜色、间距、圆角、字体、层级必须严格一致 |
| 验收标准 | 生成结果怎样才算符合设计系统 |

一个好的设计输入，不是“给 AI 更多截图”，而是让 AI 明确：

- 这是什么类型的页面
- 它由哪些稳定组件组成
- 哪些地方可以变化，哪些地方不能变化
- 生成后应该按什么标准判断对不对

## 5. 和 Agent 交互

### 5.1 如何和 Agent 交互

和 Agent 协作时，最好不要只说“帮我接入这个设计稿”。更好的方式是按阶段给信息。

### 5.2 先说明输入源和目标

```text
这是 SettingsRow 组件的设计输入。
输入源包括 Figma MCP 链接和组件截图。
目标是先生成 benchmark HTML 预览，再转换成当前项目里的 React + Tailwind 组件。
```

### 5.3 再说明组件职责和边界

```text
这个组件用于设置页列表项。
它负责展示标题、副标题、右侧值、开关或箭头。
不要把整组 Section 容器逻辑放进这个组件里。
```

### 5.4 指出关键组件和状态

```text
SettingsRow 需要支持 chevron、switch、value 三种形态。
Switch 有开启和关闭状态。
禁用状态下文字和右侧操作都需要弱化。
```

### 5.5 明确验收标准

```text
验收时重点看：
1. benchmark HTML 是否还原设计稿
2. React 组件是否有清晰 Props 和 variants
3. Storybook 是否覆盖主要状态
4. 颜色、间距、圆角是否来自 token 或统一样式规则
```

这样的交互方式能让 Agent 先建立边界，再做抽取和生成。

## 6. 产出物

### 6.1 AI / 工程会产出什么

| 产物 | 位置 | 作用 |
|---|---|---|
| React 组件 | `src/component/**` | 真实可运行的组件实现 |
| 组件 stories | `src/component/**/*.stories.tsx` | 展示状态、变体和视觉验收样例 |
| Block 模板 | `src/blocks/**` | 稳定页面或区块参考 |
| 路由索引 | `.resources/{system}/route-index.md` | prompt / 输入意图到 `page_type` 的映射 |
| 布局规范 | `.resources/{system}/layout/*.md` | 每类页面的布局骨架和生成约束 |
| Component 索引 | `.resources/{system}/components.json` | Component ID 到源码路径的映射 |
| Block 索引 | `.resources/{system}/blocks.json` | Block ID 到源码路径的映射 |
| 生成日志 | `src/render/{page-name}/index.log.md` | 记录匹配、引用、生成和验证过程 |

## 7. 最小可用接入顺序

### 7.1 最小可用接入顺序

1. 收集输入源：Figma / Pixso、设计文档、截图、旧项目代码或组件库文档。
2. 明确目标前端架构和版本号，例如 React 19 + Tailwind 4 + shadcn 4。
3. 由设计师明确组件职责、状态、变体和验收标准。
4. Agent 从组件 MCP 或截图生成 benchmark HTML 预览。
5. 设计师确认 benchmark HTML 是否贴近设计稿。
6. Agent 将 benchmark HTML 转换为 React TSX 组件源码。
7. Agent 补齐 Storybook story，覆盖主要状态和变体。
8. 设计师在 Storybook 中验收样式和状态。
9. Agent 更新 `.resources/{system}/components.json`，正式注册组件。
10. 如果组件复杂，Agent 补充 `.resources/{system}/component/{component-id}.md` 作为按需使用说明。

## 8. 验收标准

### 8.1 验收标准

组件导入完成后，不是“TSX 文件写完了”就算完成，而是要能在 Storybook 中通过真实渲染验证。

| 验收项 | 判断标准 |
|---|---|
| 架构统一 | 组件已经落到目标 React + Tailwind 架构，不再依赖原始 Figma / Vue / Angular 结构 |
| 组件真实 | 关键 UI 来自 `src/component/**`，不是临时手写相似结构 |
| 样式真实 | 颜色、间距、圆角、字体来自 token 或统一 CSS 规则 |
| 状态覆盖 | 关键组件有 stories，能看到主要状态和变体 |
| 预览对齐 | benchmark HTML 已经和设计稿大体对齐 |
| Storybook 可查 | 设计师能在 Storybook 中检查默认态、变体和交互状态 |
| 注册完成 | `.resources/{system}/components.json` 已记录组件路径、导出名和 stories |
| 使用说明 | 复杂组件有 `.resources/{system}/component/*.md` 说明 Props、变体、状态和使用边界 |
| 生成可跑 | typecheck / build / Storybook 至少有一种验证通过 |
| 设计可验收 | 设计师能判断组件是否符合设计语义和视觉规范 |

详细清单见 [资源接入验收清单](./08-资源接入验收清单.md)。

## 9. 关键原则

### 9.1 关键原则

- 输入源可以多样，但目标架构必须统一。
- Figma / 文档 / 截图表达设计意图，不直接等于生成真值。
- Vue / Angular / 旧项目组件可以提供经验，但最终要迁移为 React 真值。
- React 源码、stories、CSS、token 才是可运行真值。
- Markdown 资源层是 AI 读取源码前的地图。
- `src/render/**` 是生成结果，不能反过来作为稳定模板源。
