# 08 | 资源接入验收清单

> 这一篇用于检查一套新资源层是否真的可以被 Skill 消费。验收不是看“文件是否生成”，而是看 Component、Block、Layout、Resource Layer 是否都能追溯到真实源码，并能通过真实 prompt 跑通。

---

## 1. 验收目标

这一组先说明资源接入验收到底在验什么。

### 1.1 一句话目标

资源接入完成的标准是：

```text
设计输入已经转成真实 Component / Block / Layout 能力
  ↓
Resource Layer 能正确索引这些真值
  ↓
Agent 能根据用户 prompt 命中 page_type
  ↓
Agent 能读取 layout、Block、Component
  ↓
生成结果可运行、可检视、可追踪
```

---

### 1.2 一次完整验收包含什么

```text
输入源检查
  ↓
Component 真值检查
  ↓
Block 模板检查
  ↓
Layout / Page Type 检查
  ↓
Resource Layer 索引交叉校验
  ↓
真实 prompt 试跑
  ↓
设计师视觉和语义验收
  ↓
验收结论归档
```

验收时要记住：

```text
Resource 是地图，Source 是真值。
地图必须指向真值，不能自己变成真值。
```

---

## 2. 分阶段验收清单

这一组是实际验收时可以逐项勾选的清单。

### 2.1 输入源检查

- [ ] Figma / Pixso MCP 链接或节点 ID 正确
- [ ] 设计稿范围明确，没有误选无关 frame
- [ ] 组件、Block、页面族的命名已经初步确定
- [ ] 关键状态、变体、页面样例已经提供
- [ ] 如果输入来自截图 / 旧项目 / Vue / Angular，已说明来源和目标 React 架构

---

### 2.2 Component 真值检查

Component 是设计系统的最小真值来源。

- [ ] 关键组件已经落到 `src/component/**`
- [ ] 组件不是临时 `div` 拼出来的相似结构
- [ ] Props 命名清晰，表达组件语义
- [ ] 主要变体已经通过 `variant`、`size`、`state` 等方式表达
- [ ] 颜色、间距、圆角、字号优先使用 token 或统一样式规则
- [ ] Storybook story 覆盖主要状态和变体
- [ ] 设计师已在 Storybook 中完成视觉验收
- [ ] 已注册到 `.resources/{system}/components.json`
- [ ] 复杂组件已补 `.resources/{system}/component/{component-id}.md`

---

### 2.3 Block 模板检查

Block 是多个 Component 的稳定组合模板。

- [ ] Block 源码位于 `src/blocks/**`
- [ ] Block 复用真实 Component，没有绕过组件真值
- [ ] Block story 能在 Storybook 中渲染
- [ ] Block Markdown 位于 `.resources/{system}/block/{block-id}.md`
- [ ] Block Markdown 写清楚适用意图和不适用意图
- [ ] Block Markdown 写清楚结构树、必选区域、可隐藏区域、可替换元素
- [ ] Markdown 中声明的可隐藏区域，在 Block TSX 中有对应 prop / 配置能力
- [ ] Storybook 中能验证至少一种裁剪形态
- [ ] 已注册到 `.resources/{system}/blocks.json`
- [ ] 没有把 `src/render/**` 注册为 Block 模板源

---

### 2.4 Layout / Page Type 检查

Layout 是页面级装配能力，不只是 Markdown。

- [ ] `page_type` 命名清晰，代表一类页面
- [ ] `route-index.md` 中有对应 page_type 条目
- [ ] `layout/{page_type}.md` 存在
- [ ] layout markdown 包含 `layout_skeleton`
- [ ] layout markdown 包含 `layout_runtime` 或说明对应源码能力
- [ ] layout markdown 包含 `slots`，每个 slot 有可替换 Block 清单
- [ ] slot 可为空时明确写 `none`
- [ ] layout markdown 包含 `visibility_rules`
- [ ] layout 源码 / runtime 支持 slot 装配和显隐控制
- [ ] `reference_blocks` 都存在于 `blocks.json`
- [ ] `needed_components` 都存在于 `components.json`
- [ ] 真实 prompt 能正确命中该 page_type

---

### 2.5 Resource Layer 索引检查

- [ ] `.resources/config.json` 存在
- [ ] `active` 指向正确资源集
- [ ] `projectRoot` 指向真实 React 项目
- [ ] `.resources/{active}/route-index.md` 存在
- [ ] `.resources/{active}/layout/*.md` 存在
- [ ] `.resources/{active}/components.json` 存在
- [ ] `.resources/{active}/blocks.json` 存在
- [ ] `component/*.md` / `block/*.md` 中引用的能力不和源码冲突

---

### 2.6 源码引用交叉校验

Agent 需要做交叉校验：

| 校验项 | 通过标准 |
|---|---|
| config → projectRoot | `projectRoot` 目录存在 |
| route → layout | 每个 `page_type` 指向真实 layout 文件 |
| layout → blocks | `reference_blocks` / `slots` 中的 Block 都存在 |
| layout → components | `needed_components` 都存在 |
| layout → runtime | layout 源码或运行层能执行 slot / visibility |
| blocks → source | `blocks.json` 的 `files` 都能找到 |
| blocks → components | `dependencies` 都在 `components.json` 中 |
| components → source | `components.json` 的 `path` 和 `export` 能对应真实源码 |
| stories → Storybook | 关键 Component / Block 能在 Storybook 中预览 |
| no render source | 没有把 `src/render/**` 注册为模板源 |

---

## 3. 真实 prompt 试跑

这一组检查资源层是否真的能被 Skill 消费。

### 3.1 试跑 prompt

用一个真实页面 prompt 试跑，例如：

```text
请根据当前资源层生成一个移动端系统设置页。
要求：
- 不需要账号卡
- 需要搜索入口
- 主体包含 WLAN、蓝牙、移动网络、外观与显示分组
- 不要使用底部导航
```

这个 prompt 能同时检查：

- `route-index.md` 是否正确命中 `page_type`
- layout 是否能处理显隐规则
- slot 是否能选择正确 Block
- Block 是否能被裁剪
- Component 是否能被正确复用

---

### 3.2 检查生成过程

- [ ] 命中正确 `page_type`
- [ ] 读取了正确 `layout/{page_type}.md`
- [ ] 使用了 layout 中的 fixed_blocks / slots
- [ ] 按要求隐藏了不需要的区域
- [ ] 没有临时发明未注册 Block
- [ ] 使用的 Component 都来自 `components.json`
- [ ] 使用的 Block 都来自 `blocks.json`
- [ ] 需要裁剪 Block 时，参考了对应 `block/*.md`
- [ ] 生成结果没有从 `src/render/**` 复制历史产物

---

### 3.3 检查生成产物

- [ ] 生成页面源码
- [ ] 生成 Storybook story 或可预览入口
- [ ] 生成记录说明使用了哪个 page_type、layout、Block、Component
- [ ] TypeScript / build / Storybook 至少有一种验证通过
- [ ] 页面截图或预览可正常渲染

---

## 4. 设计师验收

这一组给设计师做最终视觉和语义判断。

### 4.1 视觉验收

- [ ] 页面结构符合设计稿或页面族规范
- [ ] 使用了正确组件，不是临时手写相似结构
- [ ] Block 组合符合预期
- [ ] 可隐藏区域没有留下异常空白
- [ ] slot 替换后的视觉仍然稳定
- [ ] 颜色、间距、字号、圆角、阴影符合设计系统
- [ ] 长文本、缺图标、空数据等边界不破版

---

### 4.2 语义验收

- [ ] 页面类型匹配正确
- [ ] 页面意图没有被误解
- [ ] 组件状态和变体覆盖完整
- [ ] Block 的适用场景正确
- [ ] Layout slot 替换合理，没有过度泛化
- [ ] 不应该出现的区域已经隐藏
- [ ] 复杂组件 / Block 的使用方式符合 Markdown 说明

---

## 5. 验收结论

这一组用于记录验收结果，方便后续维护和回归。

### 5.1 通过标准

资源接入可以通过验收，必须同时满足：

- Component 真值通过 Storybook 验收。
- Block 源码和 Block Markdown 都完整。
- Layout markdown 和 layout runtime 能力一致。
- Resource Layer 索引能完整指向 Source Layer。
- 真实 prompt 试跑能生成正确页面。
- 设计师视觉和语义验收通过。

---


---

### 5.3 不通过时怎么处理

| 问题类型 | 回到哪一步修 |
|---|---|
| 组件样式或状态不对 | 回到 `04-抽取 Component 资源` |
| Block 结构或裁剪不对 | 回到 `05-抽取 Block` |
| slot / 显隐 / page_type 不对 | 回到 `06-配置 Page Type 与 Layout` |
| 索引路径或依赖不对 | 回到 `07-生成 Resource Layer 索引` |
| 真实 prompt 命中错误 | 检查 `route-index.md` 和 layout |
| 生成页面视觉不对 | 先判断是 Component、Block 还是 Layout 问题 |

如果验收失败，不要直接改生成页面结果；应该回到对应真值层修复，再重新试跑。
