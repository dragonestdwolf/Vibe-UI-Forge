# 01 | 为什么是这样一个架构

> 这一篇回答核心问题：为什么需要这套 Skill，以及它为什么采用当前这种架构设计。

---

## 1. 为什么普通 AI 生成不稳定

### 1.1 问题的本质

当我们把设计稿直接交给 AI 生成页面时，会遇到一个根本问题：**AI 生成的 UI 不够规范、不够稳定**。

具体表现为：

| 问题 | 例子 |
|------|------|
| 同一组件每次生成结构不同 | 这次生成的 Button 用 `div`，下次用 `button` |
| 样式与设计系统不一致 | 用了 `bg-blue-500` 而不是系统的 `bg-primary` |
| 组件之间配合混乱 | Dialog 的按钮样式和表单的按钮样式完全不同 |
| 无法预测生成结果 | 同一个页面让 AI 生成两次，结果差异很大 |

**根本原因：AI 生成时没有足够的约束。** 它不知道设计系统里组件的 Props、变体、状态、样式规范，也不知道一个页面应该优先复用哪些稳定结构。

### 1.2 解决方案：向 AI 注入规范内容

要让 AI 稳定地生成符合设计系统的 UI，需要在生成过程中**向 AI 注入规范内容**，包括组件的 Props、变体、状态，以及页面结构模板。

注入的方式不同，就产生了不同的方案。我们对比了三种主流方案，并在真实项目中验证过：

| 方案 | 注入什么 | 验证结论 |
|------|---------|---------|
| 纯 Markdown | 设计说明文档 | 信息密度低，AI 容易漏掉关键约束，生成结果碎片化 |
| Markdown + CSS/HTML 片段 | Tailwind 片段、HTML 模板 | 真实 B 端页面片段过多，容易撑爆模型上下文，注意力涣散 |
| 生产级 React 组件库 | 真实可运行的组件源码 | **唯一能达到生产级稳定性的方案** |

---

## 2. 为什么最终需要真实 React 组件库

### 2.1 Markdown 和 CSS/HTML 片段的局限

**Markdown 的问题**：AI 的注意力是有限的。一段 200 行的 Markdown 描述，AI 实际能精准记住的关键信息可能不到 30%。

更重要的是，**Markdown 无法验证理解**。文档说“这个按钮支持 disabled 状态”，但 AI 不知道 `disabled` 是 prop、className，还是 CSS 变量。

**CSS/HTML 片段的问题**：CSS 是无状态的描述，不包含组件的行为约束。更致命的是，在真实 B 端项目里，局部片段会很快变成上下文噪音：

```text
一个 B 端表格页面可能包含 20+ 个组件的样式片段
每个片段 5-20 行 CSS/HTML
总上下文 100-400 行代码片段
         ↓
直接撑爆模型上下文
注意力被碎片化的代码片段分散
无法聚焦在真正的业务逻辑上
```

### 2.2 生产级 React 组件库为什么更稳

React 组件库把视觉、结构、状态和交互封装成可复用 API。

```tsx
import { Button } from "@/component/Button"

<Button variant="primary" disabled>
  提交
</Button>
```

这一行 import 背后包含：

- **Props 定义**：AI 知道有哪些属性可用
- **变体约束**：AI 知道什么能做、什么不能做
- **状态处理**：AI 知道组件有哪些交互状态
- **样式来源**：AI 知道用什么 token
- **可运行代码**：组件通过了测试，AI 生成结果的行为是可预期的

AI 在 Source Grounding 阶段直接读取组件源码，理解组件的完整语义。生成的结果天然会更接近设计系统，而不是临场拼样式。

### 2.3 核心结论：资源转换无法逃避

**不是说 shadcn 的项目架构有多好，而是说必须有一份完整、成熟的生产级前端组件库，才能真正支撑页面生成。**

| 需求 | 为什么必须满足 |
|------|---------------|
| AI 能读取组件源码 | AI 知道组件的 Props、变体、状态 |
| 组件行为有完整约束 | AI 生成的结果是可预期的 |
| 安装流程可预测 | AI 知道组件在哪里、如何引用 |
| 与样式系统一致 | AI 生成的样式和组件原生风格一致 |

我们尝试了多种方案，最后发现：在生产级项目里，资源转换无法逃避“必须有一份真实 React 组件库”这个事实。

本项目借用 shadcn 的思路：组件源码放在项目里、组件 API 可被 AI 读取、`components.json` 提供项目信息、`.agent/skills/shadcn` 定义通用生成规则。**但本项目不是只服务 shadcn**，只要你的设计系统能提供真实 React 组件、Block、token 和资源契约，也可以接入。

## 3. 设计师在这套架构里负责什么

### 3.1 设计师的职责

设计师负责定义“设计语义”，而不是手写代码：

- **页面族**：哪些页面属于同一类
- **页面类型**：这个页面应该叫什么（page_type）
- **布局意图**：顶部、主体、卡片、列表如何组织
- **组件语义**：有哪些状态、变体、使用边界
- **验收标准**：生成结果怎样才算符合设计系统

AI / 工程负责把这些语义落成文件：React Component / Block 源码，以及 `.resources/{system}/` 下的 `route-index.md`、`layout/*.md`、`blocks.json`、`components.json`。

### 3.2 判断准则

| 如果你只有 | AI 会怎样 |
|-----------|----------|
| 截图 | 猜布局、猜组件、猜样式 |
| Markdown | 读懂大意，但细节不稳定 |
| CSS/HTML 片段 | 能复用局部，但上下文容易爆炸 |
| React 组件库 + 资源层 | 有源码真值，也有语义地图 |

**所以资源接入的目标不是“写更多设计说明”，而是让设计说明、React 源码和资源索引互相对齐。**
