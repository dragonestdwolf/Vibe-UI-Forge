# Wiki 目录重构计划 v2：按读者视角重组知识结构

## 背景

当前 wiki 已经有比较完整的内容骨架，但目录名偏工程动作，例如“工作流程”“维护指南”。这些名字对第一次接触项目的平台设计师不够直观：

- “工作流程”不明确是谁的工作流程：AI、设计师还是工程？
- “维护指南”不明确维护的是资源层、组件库、页面模板还是生成结果。
- 从 Figma 开始接入资源时，不容易判断应该先看哪篇。
- Skill 原理、资源接入、后续维护被拆散在多个目录中，阅读路径不够连续。

因此建议按设计师理解项目的生命周期重组：

```text
先理解 Skill 是什么
  ↓
再从 Figma / MCP / React 组件库构建资源接入
  ↓
最后学习后续如何修改和维护
```

## 重构目标

将 wiki 从“工程文档集合”整理成“设计师接入手册 + 工程参考资料”。

新的信息架构按不同视角区分大模块：

1. **认识 Skill**：解释这套系统是什么、为什么需要 React 组件库 + Markdown 资源契约、Skill 如何工作。
2. **从 0 构建资源接入**：解释设计师如何从 Figma / MCP 出发，把设计稿转成 React 组件、Block、Layout、Resource Layer。
3. **后续修改与维护**：解释资源接入后，如何判断改 CSS、改 Component、改 Page/Block。
4. **参考资料**：集中放字段模板、资源层文件说明和术语表。

## 核心主线：从 Figma 到 Resource Layer

新 wiki 的核心主线应该是一条连续链路：

```text
Figma 设计稿
  ↓ MCP 读取
设计师定义页面族、组件语义、状态
  ↓
AI 生成 React Component / Block
  ↓
工程验证组件是否真实可运行
  ↓
抽取 components.json / blocks.json
  ↓
编写 route-index.md / layout markdown
  ↓
形成可被 Skill 消费的 Resource Layer
  ↓
生成页面并验收
```

这条主线需要明确分工：

| 阶段 | 设计师负责 | AI / 工程负责 |
|---|---|---|
| Figma 读取 | 提供 Figma 链接、页面范围、目标设计系统 | 通过 MCP 读取节点、截图、变量和组件信息 |
| 组件语义整理 | 定义组件用途、状态、变体、使用边界 | 生成或整理 React 组件代码和 stories |
| Page Type 抽取 | 归纳页面族、命名页面类型、定义命中和排除规则 | 写入 `route-index.md` |
| Layout 抽取 | 定义页面骨架、区块顺序、关键约束 | 写入 `layout/{page_type}.md` |
| Component / Block 索引 | 审核组件和区块是否表达设计语义 | 写入 `components.json`、`blocks.json` |
| 验收 | 检查设计语义、视觉一致性、状态覆盖 | 跑构建、截图、生成日志 |

## 重构原则

- 不改变 `.resources` 的现有 contract，只改变 wiki 的表达方式。
- 不要求设计师手写复杂 JSON；JSON 模板用于审阅和沟通。
- 不把 `src/render/**` 当作模板源；它只能作为生成结果预览。
- 真实生成依据仍然是 React 源码、stories、CSS 和 token。
- 文档要优先回答设计师的问题：我现在处在哪一步？我需要提供什么？我怎么验收？

## 建议实施步骤

### Step 1：先建新目录，不删除旧目录

创建 `01-认识 Skill`、`02-从 0 构建资源接入`、`03-后续修改与维护`、`04-参考资料` 四个目录。

先把旧内容复制或移动到新目录中，保持旧目录暂时不删，降低迁移风险。

### Step 2：改写 `00-快速入口.md`

将入口从“按工程动作”改成“按读者任务”：

| 我想做什么 | 去哪里 |
|---|---|
| 第一次了解项目 | `01-认识 Skill` |
| 从 Figma 接入设计系统 | `02-从 0 构建资源接入` |
| 修改样式、组件或页面 | `03-后续修改与维护` |
| 查字段、模板、术语 | `04-参考资料` |

### Step 3：迁移并合并内容

按迁移映射表整理旧文档内容。重复内容合并，过时链接删除，占位的“待补充内容”替换成真实说明。

### Step 4：补齐从 0 接入主线

重点补写 `02-从 0 构建资源接入` 下的 8 篇文档，确保设计师能从 Figma 一路理解到资源层验收。

### Step 5：清理旧目录

确认新目录可读后，再考虑删除旧目录或保留跳转页。删除前需要检查 wiki 内链，避免断链。

## 验收标准

- 设计师能通过 `00-快速入口.md` 判断自己应该看哪一组文档。
- 新目录名直接表达读者任务，不再依赖“工作流程”“维护指南”这类模糊词。
- 从 Figma 到资源层的接入路径是一条连续链路，而不是分散在多篇工程说明里。
- CSS / Component / Page-Block 的维护边界仍然保留，并放在“后续修改与维护”模块。
- 所有模板和术语集中在“参考资料”模块，减少主流程文档的认知负担。

## 暂不执行的事项

- 本计划只描述目录重构，不立即移动文件。
- 不修改 `.resources`、React 源码或 shadcn skill。
- 不重新设计资源层 schema。
- 不删除旧目录，除非后续单独确认。
